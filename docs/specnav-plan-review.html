产品定位

SpecNav 不再被定义为单个测试插件，而是构建在 OpenSpec 之上的 Claude Code 工程生命周期插件套件。

已确认

全局失败策略

SpecNav 对必需工作流状态没有 fallback。缺依赖、缺 OpenSpec、缺 hook、缺 artifact、缺 context manifest、验证工具失败，都必须明确报告 blocker，并阻断依赖动作。

允许继续的非生产动作

  • 只读 status / doctor 检查。
  • OpenSpec / SpecNav 初始化动作。
  • 创建或修复 openspec/** 工作流 artifact。
  • 不触碰生产代码的文档编辑。

必须阻断的动作

  • 缺 openspec/ 时写生产代码。
  • 缺 scope.json 时写生产代码。
  • 缺 context manifest 时派发对应 subagent。
  • 验证证据过期或缺失时 archive / release。

需求阶段前置四件套

需求阶段采用 spec-gated grilling：先验证项目级基础规格，再进入一问一答的需求澄清。缺任意一个基础 spec 时，SpecNav 不能继续问功能需求，只能引导用户创建或修复对应文档。

1

UI Design Spec

openspec/specs/ui-design/design.md

采用 Geist 风格结构：YAML token contract 加 Markdown usage guide，覆盖 colors、typography、spacing、rounded、components、motion、voice。

2

System Architecture & Database Spec

openspec/specs/system-architecture/design.md

定义系统拓扑、模块边界、前后端架构、API、数据库实体、权限、安全、集成和运维约束。

3

Frontend-Backend Data Flow Spec

openspec/specs/frontend-backend-data-flow/design.md

定义用户动作到前端状态、请求、后端服务、数据库读写、响应、渲染、错误、重试和回滚的完整链路。

4

Component Architecture & Reuse Spec

openspec/specs/component-architecture/design.md

强制高内聚低耦合，定义组件分类、公共组件抽离、public API、状态归属、组合模式、命名和测试要求。

核心规则：基础 spec owns project-level constraints。需求文档只能引用、压力测试或提出变更，不能在 change artifact 里临时发明 UI 规则、架构边界、隐藏 payload、隐式校验、数据库副作用或组件抽象。

缺失时的强制引导

  • 缺 UI spec：按用户提供的 Geist design.md 格式，引导创建 YAML token contract 和 Markdown guide。
  • 缺架构 spec：先问系统拓扑、模块边界、数据库模型、权限安全、运维约束。
  • 缺链路 spec：逐条补用户触发、前端状态、请求响应、后端服务、数据库读写、错误和回滚。
  • 缺组件 spec：补组件分类、高内聚、低耦合、抽离阈值、public API、状态归属和测试要求。

问需哲学

  • 先读 specs 和代码证据，不问已经能从项目里确认的问题。
  • 一次只问一个关键问题，每个问题带推荐答案和取舍说明。
  • 每个分支必须闭合成可执行决策，再进入下一题。
  • 输出 requirements.md、acceptance.md、spec-map.json、component-impact-map.json。
组件抽离触发 SpecNav 行为
两个以上位置出现相同 UI 结构、交互、表单组、状态展示、动作组、数据转换或 API 编排。 必须抽离公共 component、hook、utility 或 service，不能复制粘贴。
组件混合渲染、数据请求、校验、业务规则，超出声明职责。 必须拆分职责并记录到 component-impact-map.json。
只有一个使用点，或相似代码代表不同业务概念。 不得强行抽象，避免泛化 props 掩盖领域语义。

原型阶段

Prototype 阶段不是只写说明，也不是直接开始生产实现。它必须产出隔离、可运行、可审查的原型代码，让用户能直观看到或跑到关键形态，然后再决定是否进入开发。

核心规则：Prototype code is required for review, but it is not production code. It must be isolated, labeled, runnable, reviewable, and promotable only through development gates.
Gate

入口条件

  • 四个 foundation specs 有效。
  • requirements.md、acceptance.md 已存在。
  • spec-map.json 和 component-impact-map.json 无 unresolved gaps。
  • 必要的设计系统、UI kit、组件、截图、品牌、API 或领域上下文已找到。
Code

原型代码必须存在

  • UI 原型：prototype/artifact/index.html。
  • 逻辑原型：prototype/logic/。
  • API 原型：prototype/api/。
  • 组件边界原型：prototype/component/。
Boundary

生产隔离

  • 原型代码不能进入生产路径。
  • 不能直接复制成生产实现。
  • 转正必须经过 scope.json、tasks.md、组件抽离、契约和验证门禁。
问题类型 Prototype branch 原型输出
UI、视觉、布局、交互 ui-html 高保真 HTML，包含 variants、tweaks、screen-map。
状态机、业务逻辑、后端行为 logic-state 可运行逻辑原型，每次操作后暴露完整状态。
API 或数据契约 api-contract schema、mock server、请求响应和错误样例。
前后端完整链路 data-flow 可观察的 flow harness 或链路演示。
组件/API 边界 component-seam 组件树、public API、可复用点和抽离影响。

从参考仓库吸收

  • Matt：原型必须回答一个明确问题，完成后沉淀 learned 和 handoff。
  • Superpowers：可视化原型 just-in-time 出现，先设计审批再进入实现。
  • Trellis：prototype 是 workflow state，由状态面包屑防止跳到 development。
  • Aegis：用 Requirement Ready、BaselineUsage、Product Risk、Architecture Integrity 检查方向。

审批与交接

  • prototype-manifest.json 声明类型、入口、依赖、mock、真实数据、promotion 策略。
  • decision.json 必须为 approved 才能进入 development。
  • handoff.md 固化 approved variant、要实现的屏幕、组件、API、data flow、状态和测试。
  • 开发阶段只能实现 handoff 里批准的内容。

开发阶段

Development 不是发明决策的地方，而是把已经批准的需求、原型和基础规格，通过受控的垂直切片转成生产代码。

核心规则：No production code without approved requirements, approved or explicitly skipped prototype, valid scope.json, vertical-slice tasks, task brief, TDD evidence, spec review, and quality review.
Gate

入口条件

  • 四个 foundation specs 有效。
  • requirements.md、acceptance.md 已存在。
  • spec-map.json、component-impact-map.json 无 unresolved gaps。
  • prototype/decision.json 为 approved，或明确 not_required 且有原因。
Scope

scope.json 锁定

  • 缺 scope.json 阻断生产编辑。
  • 超出 allowed_roots 阻断。
  • 命中 denied_roots 阻断。
  • 删除、迁移、依赖和 shared component 变更升级 review 强度。
Slice

垂直切片任务

  • 禁止按 database/API/UI 分层拆任务。
  • 每个 slice 必须窄而完整。
  • 必须可独立验证或演示。
  • 必须声明非目标、允许文件、接口 seam 和 focused tests。
Artifact 用途 阻断规则
development/before-dev-check.json 记录开发入口门禁的确定性检查结果。 任一上游 artifact 缺失或无效时阻断。
development/prototype-promotion-map.json 声明原型哪些决策可迁移、哪些代码必须重写。 缺失时禁止从原型转入生产实现。
development/tasks/*/brief.md 单个垂直切片的目标、范围、允许文件、接口、状态和测试要求。 缺 brief 或 brief 不完整时不能派发 implement。
development/tasks/*/spec-review.md 确认实现是否准确覆盖需求、验收、prototype handoff 和 specs。 未 approved 时禁止进入 quality review。
development/tasks/*/quality-review.md 检查高内聚低耦合、测试质量、错误处理、复用和复杂度。 未 approved 时禁止任务完成。
development/drift-check.jsonl 记录实现是否偏离 requirements、prototype handoff、spec map 和 component map。 blocking drift 存在时禁止交接验证。

原型转正规则

  • 可以迁移 approved copy、token names、state names、interaction names。
  • 必须重写 component structure、state management、API calls、validation logic、styling implementation。
  • 禁止直接复制 inline prototype JS、mock data、prototype CSS reset、temporary helpers。
  • 生产约束和原型冲突时，回到 prototype 或 foundation spec，不在开发阶段临时决定。

实现与 Review Loop

  • 实现状态只允许 DONE、DONE_WITH_CONCERNS、NEEDS_CONTEXT、BLOCKED。
  • DONE 进入 spec review，NEEDS_CONTEXT 更新 brief/context 后重派。
  • 顺序固定：implementer report -> spec review -> fix -> spec re-review -> quality review -> fix -> quality re-review -> ledger complete。
  • review 结论必须来自文件、测试、日志、生成 artifact 和运行态，不能只凭聊天记忆。

必需

context/*.jsonl

journal/

人类可读项目记忆，记录会话摘要、决策、blocker、验证结果和运维交接。

验证阶段

Verification 是完整测试阶段，只能在 development 产生 handoff-to-verify.md、task ledger 完成、spec/quality review approved、drift 清零之后启动。

核心规则：六个验证域都必须有独立 skill、独立证据和独立 report。任何 required domain 缺失、失败或 blocked，aggregate report 都是 red，不允许 fallback。
Plan

verify/plan.json

基于 changed files、risk tier、requirements、prototype handoff、component impact 和 development handoff 生成验证计划。

Evidence

evidence-index.jsonl

索引每个命令、文件、日志、截图、report 和人工审查记录，支撑六个 domain 的结论。

Gate

aggregate-report.json

归档和运维交接的机器门禁。任一 domain red 或 blocked，聚合结果必须 red。

Trace

traceability-matrix.json

每个 changed file 都必须映射到 requirement、task、prototype decision、foundation spec 和验证域；unmapped changes 直接 red。

Receipt

receipt.json

记录检查动作、结果、覆盖范围、未覆盖范围、残余风险和信心等级，防止 green report 被说成无限完成。

Eval

behavior-evals/

把 skill 行为评测和普通脚本测试分离，用 clean-session transcripts 验证 bootstrap、router 和 gate 行为。

Skill 验证目标 必需证据 主要阻断项
verify-facticity 审计 specs、架构声明、API、依赖、配置、路由、数据库声明、traceability 和生成 artifact 是否符合真实系统。 claims.jsonl、repo-inventory.json、traceability-matrix.json、文件或命令引用。 虚构、过期、未文档化或 unmapped 事实。
verify-static OpenSpec validate、lint、type check、依赖检查、schema 校验、禁用模式和结构扫描。 命令日志、退出码、配置文件、静态报告。 语法、风格、类型、schema 或 policy 失败。
verify-unit 验证变更逻辑有 focused unit/regression coverage，并检查测试是否通过 public interface 验证真实行为。 test-map.json、red/green 输出、test-quality-rubric.json、coverage notes、显式不可测例外。 核心逻辑未测、测试失败或测试耦合实现细节。
verify-redteam 破坏性、对抗性、注入、边界、权限、滥用和韧性测试。 threat model、probes、payloads、预期与实际结果，失败时写入 root-cause check。 可利用安全或鲁棒性问题。
verify-e2e 验证完整用户/业务链路，覆盖前端、后端、API、状态、数据库和外部集成路径。 flow list、run logs、必要时截图或 trace。 集成链路断裂或状态不一致。
verify-sensory 人工 UX / Code Review，覆盖可读性、交互质量、性能体感、可访问性、维护性、reviewer independence 和用户体验。 reviewer-independence.md、review notes、findings、必要时截图或人工检查记录。 人类体验、维护性或 review 独立性不可接受。

失败路由

  • facticity 文档失败：回 requirements 或 foundation spec。
  • static / unit 失败：回 development fix 或 task split。
  • redteam 失败：按根因回 requirements、architecture、data-flow 或 development。
  • e2e 失败：回 data-flow spec、prototype 或 development。
  • sensory UX 失败：回 prototype 或 UI design spec。
  • sensory code-quality 失败：回 development、component spec 或 refactor task。

Blocker 分类

  • tool-unavailable：工具或命令不存在。
  • env-auth：认证、账号、凭据不足。
  • env-runtime：运行时、provider、浏览器、服务条件不足。
  • contract-regression：插件或宿主契约退化，直接 red。
  • insufficient-evidence：证据不足，不能降级成 warning。
  • product-ambiguity / scope-drift：回需求或开发范围。

测试质量 Rubric

  • 测试必须验证用户或 caller 能观察到的行为。
  • 测试必须通过 public interface，不能绕进私有实现。
  • 测试应能承受内部重构。
  • 内部 mock、私有方法、调用次数/顺序断言是 red flag。
  • 无法通过稳定 interface 测试时，回 component architecture 或 development 修正。

Reviewer 独立性

  • reviewer 只读，不修改工作区、index、branch 或待审 artifact。
  • implementer report 和 controller summary 只是 claim。
  • controller 不得要求 reviewer 忽略、降级或预设 finding。
  • 无法从证据验证时写 cannot verify，由 controller 补证据。

Domain Report Contract

  • 每个 domain 只允许 green、red、blocked。
  • 每个 report 必须列 Inputs Reviewed、Evidence、Commands Run、Findings、Required Fixes、Residual Risk。
  • blocked report 必须写入 blocker_class。
  • Required fixes 阻断 aggregate green。
  • Residual risk 只有在显式、非阻断且被 sign-off policy 接受时才允许存在。

完成门禁

  • verify/plan.json 存在，所有 planned domains 已运行。
  • traceability-matrix.json 没有 unmapped changes。
  • 六个 domain reports 都存在且 required domain 全部 green。
  • evidence-index.jsonl 覆盖每个 verdict 的证据。
  • receipt.json 明确 covered scope、uncovered scope、residual risk 和 confidence。
  • blocker-classification.jsonl 没有 unresolved blocker。

  • 高风险变更已有 required sign-off。

Target 语义 必需 artifact
local-only 保留本地变更，交给用户控制 merge 或 install。 readiness、branch finish、必要 changelog。
plugin-marketplace 作为 Claude Code plugin 发布或安装。 release checklist、install verification、compatibility matrix、README 更新。
package 产出版本化本地包，例如 zip 或 tarball。 release notes、package validation、可选 checksum。
host-compatibility 声明或更新某个宿主支持状态。 compatibility matrix、宿主 install verification、known limitations。
project-deploy 部署应用或运行时变更。 deploy plan、rollback plan、monitor plan、postmortem template。

Install Verification

  • 必须区分 marketplace_root 和 plugin_root。
  • 多插件仓库中 SpecNav 的 plugin_root 由 marketplace.json.plugins[].source 解析，不能假设仓库根就是插件根。
  • 必须从 plugin root / method-pack root 执行 doctor，不从 target project 执行。
  • JSON 成功字段必须明确：ok、workspaceSupport、configStatus。
  • 宿主有独立 discovery root 时必须额外验证。
  • 缺 hook、skills、commands、OpenSpec 或 context manifests 时不能发布。

Update Policy

  • 默认只更新当前宿主。
  • 更新所有宿主必须有用户显式 all-host 请求。
  • 共享 plugin root 可以只更新一次，但每个宿主 exposure 都要重新同步并 doctor。
  • 不允许后台静默自动更新。

Branch Finish

  • 先验证测试，再给 merge、PR、keep、discard 选项。
  • 用只读 git 命令记录 git-dir、git-common-dir、branch 和 worktree。
  • PR 和 keep 默认保留 worktree。
  • 只清理有 SpecNav provenance 的 worktree。

Deploy

  • 记录环境、目标、命令、配置、secret 前提。
  • 记录数据库或 migration 影响。
  • 列出 smoke checks、owner 和 deploy window。

Rollback

  • 部署前必须存在 rollback plan。
  • 记录触发条件、回滚命令、数据恢复或 migration reversal。
  • 没有可行 rollback 时必须写入 explicit risk acceptance。

Monitor

  • 记录日志、指标、端点、队列或用户链路。
  • 记录观察窗口、正常值、负责人和升级路径。
  • 监控不可用时不能假装部署完成。

Postmortem 与 Writeback

  • 验证或部署出现 blocking failure、rollback、安全/数据/可用性风险、重复失败时必须写 postmortem.md。
  • update-spec 检查学习是否需要回写 requirements、UI design、architecture、data-flow、component architecture、runbook 或 known limitations。
  • 有学习但未回写或未显式 deferred 时阻断 archive。

完成门禁

  • readiness.json.ready 为 true。
  • release target 所需 artifacts 全部存在。
  • 插件面变更时 install verification 通过。
  • 宿主支持声明变化时 compatibility matrix 已更新。
  • 部署范围内存在 rollback 和 monitor plan。
  • branch finish 已记录或明确 not applicable。
  • 用户可见变更有 changelog 和 release notes。
  • archive-gate.json.verdict 为 green。

十个治理点

1

Bootstrap / Router

using-specnav 在 SessionStart 注入，让模型每次先检查 SpecNav/OpenSpec 状态并按合法动作路由。

2

状态机与 Artifact Contract

每个阶段声明必需文件、合法动作、阻断动作、修复动作、context manifests 和验证域。

3

Project Memory / Journal

补充人类可读 journal，避免只依赖机器事件日志。

4

Spec Update / Learning Writeback

update-spec 在归档或发布前回写新约束、新接口、新风险和测试经验。

5

Doctor / Health Check

/specnav-doctor 检查插件安装、hooks、OpenSpec、skills、agents、commands 和状态一致性。

6

Context Manifest / Subagent Injection

用各阶段 JSONL manifest 精准注入上下文，缺失则阻断派发。

7

Debug / Break-loop

重复失败时强制进入根因分析，记录复现、假设、失败尝试和下一步实验。

8

Git / Worktree / Branch Finish

将分支收尾和 OpenSpec archive 分离，防止误清理用户工作区。

9

Release / Distribution / Update

补 release checklist、plugin validation checklist、compatibility matrix、双语 README。

10

Behavior Evals / Clean-session Acceptance

用干净会话验证真实模型行为，而不是只测脚本。

待确认问题

OpenSpec 变更创建方式

/specnav 应直接创建 OpenSpec change，还是只调用 OpenSpec 官方命令？建议优先走 OpenSpec CLI，避免 SpecNav 成为第二套状态源。

高风险升级时机

高风险 tier 应在 PreToolUse 即时升级，还是只在 archive/release 前检查？建议 PreToolUse 先标记，archive/release 强制执行。

Sign-off 来源

默认使用本地 signoff.yaml，还是接 PR review 元数据？建议 MVP 用 signoff.yaml，后续增加 PR adapter。

SAST 适配器顺序

先接哪个安全扫描工具仍未定。建议先做通用 command adapter，再接具体工具。

Context manifests 生成源

由 OpenSpec state、SpecNav scripts，还是 combined schema 生成？建议 SpecNav scripts 生成，OpenSpec artifacts 作为权威输入。

Behavior eval harness

clean-session acceptance 用哪套 harness 先落地？建议先用 Claude Code 本地脚本化 smoke，再评估是否引入更完整的 LLM eval。

首个 Release Target

先实现 plugin marketplace、local package，还是 host compatibility smoke？建议先做 local package 加 Claude Code install verification，再扩展宿主矩阵。
